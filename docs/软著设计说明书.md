## 2 软件概述

### 2.1 软件背景

随着企业数字化转型的深入，身份认证与访问控制已成为各类应用系统的核心安全基础。在 Java 生态中，Sa-Token 等框架为企业提供了完整的会话治理和权限控制能力。然而，在 Node.js / NestJS 生态中，缺乏一款能够同时覆盖登录认证、会话管理、RBAC 权限、OAuth2 / OIDC、SSO 单点登录、API 签名、微服务鉴权等全场景的企业级认证框架。

本软件正是在这一背景下设计和开发的。本软件参考 Sa-Token 的会话治理理念，专为 NestJS 生态打造，旨在为开发者提供一套类型安全、模块可插拔、功能完整的统一身份认证与授权解决方案。

### 2.2 软件目标

本软件的主要目标包括：

1. 为 NestJS 应用提供开箱即用的登录认证与授权能力，减少业务系统重复开发安全模块的成本。
2. 支持 JWT 与随机 Token 双策略，满足不同场景对状态无感知或有状态会话的需求。
3. 提供企业级会话治理能力，包括单点登录、多点登录、互斥登录、强制下线、踢人、封禁、身份切换、滑动续签等。
4. 内置 RBAC 权限引擎，支持角色、权限及通配符匹配。
5. 实现 OAuth2 / OIDC 授权服务器，支持标准授权模式和安全扩展。
6. 支持跨域 SSO 单点登录，满足多系统统一认证需求。
7. 提供 API 签名认证、登录限流、设备指纹、分布式锁等安全加固能力。
8. 支持 WebSocket 认证、微服务 gRPC / TCP 鉴权，覆盖 HTTP 之外的多协议场景。
9. 提供审计日志、多账号切换、Remember Me 等辅助功能。
10. 通过抽象持久化接口和存储适配器，支持 MySQL、PostgreSQL、Oracle、SQLite、MongoDB、内存等多种存储后端。

### 2.3 软件定位

本软件是一款面向 NestJS 生态的企业级认证授权框架，主要面向 Web 应用、移动应用后端、SaaS 平台、微服务架构以及需要统一身份治理的企业信息化系统。开发者可以通过简单的模块注册和装饰器使用，将本软件集成到现有 NestJS 项目中，无需从零实现复杂的认证逻辑。

### 2.4 软件运行环境

#### 2.4.1 硬件环境

- 处理器：x86-64 或 ARM64 架构，主频 1.8GHz 及以上
- 内存：最低 2GB，建议 4GB 及以上
- 存储：最低 10GB 可用空间
- 网络：支持 TCP / IP 协议栈

#### 2.4.2 软件环境

- 操作系统：Linux、Windows、macOS 等支持 Node.js 运行的操作系统
- 运行时：Node.js 22 及以上版本
- 框架依赖：NestJS 11 及以上版本
- 可选中间件：Redis 5.0 及以上版本（用于分布式会话、限流、分布式锁、OAuth2 存储等）
- 可选数据库：MySQL 8.0、PostgreSQL 13、Oracle 12c、SQLite 3、MongoDB 4.4 及以上版本（用于持久化适配器）

## 3 系统总体设计

### 3.1 设计目标

本软件的总体设计目标是在保证安全性的前提下，实现高内聚、低耦合、易扩展的认证授权框架。具体目标如下：

1. **模块化设计**：将不同功能封装为独立模块，用户可按需引入，避免不必要的依赖膨胀。
2. **类型安全**：基于 TypeScript 原生类型系统，提供完整的类型推断和编译期检查。
3. **框架原生集成**：以 NestJS 动态模块、装饰器、守卫、拦截器等原生机制为基础，无需额外适配层。
4. **可插拔存储**：通过抽象接口支持内存、Redis、SQL 数据库、NoSQL 数据库等多种后端。
5. **协议兼容**：遵循 OAuth2、OIDC、JWT 等行业标准协议，保证互操作性。
6. **安全加固**：在关键路径使用常量时间比较、Nonce 去重、PKCE、Refresh Token Rotation 等机制。

### 3.2 设计原则

1. **单一职责原则**：每个模块只负责一类认证相关职责，如登录认证、权限控制、OAuth2、SSO 等。
2. **开闭原则**：通过接口和抽象类定义扩展点，新增功能时尽量不改既有代码。
3. **依赖倒置原则**：核心模块依赖抽象接口，具体实现由适配器提供。
4. **最小惊讶原则**：API 命名和行为符合 NestJS 社区习惯，降低学习成本。
5. **默认安全原则**：安全相关配置默认采用较严格的值，如 Cookie 默认 secure 为 true。

### 3.3 技术架构

本软件采用分层架构设计，从上到下依次为：

#### 3.3.1 接入层

接入层负责与外部请求进行交互，包括 HTTP 请求、WebSocket 连接、gRPC / TCP 微服务调用。本软件通过 NestJS Guard、Interceptor、Filter 等机制，在接入层完成身份校验、权限校验、异常统一处理。

#### 3.3.2 核心服务层

核心服务层是本软件的业务逻辑中枢，包括：

- AuthService：处理登录、登出、Token 校验、会话管理、二级认证等核心操作。
- AuthGuard：全局认证守卫，根据 Token 判断用户是否已登录，并执行角色权限校验。
- PermissionEngine：RBAC 权限引擎，执行角色与权限的匹配计算。
- SessionStore：会话存储抽象，管理用户会话的增删改查及过期清理。

#### 3.3.3 策略与扩展层

策略与扩展层提供可替换的实现和可选功能：

- TokenStrategy：JWT 策略、随机 Token 策略。
- StoreAdapter：MemoryStore、RedisStore。
- PersistenceAdapter：SQL、MongoDB、内存适配器。
- PasswordEncoder：BCrypt、Argon2 加密器。
- 可选扩展模块：OAuth2Module、SsoModule、MicroserviceModule、SignatureGuard、RateLimiter、DistributedLock、AuditService 等。

#### 3.3.4 基础设施层

基础设施层负责与外部中间件和存储系统交互，包括 Redis 客户端、数据库驱动、日志输出、文件系统等。

### 3.4 模块划分

本软件由核心认证模块和可选扩展模块两大部分组成：

#### 3.4.1 核心认证模块（auth）

核心模块包含以下子模块：

- 认证服务（auth.service）
- 认证守卫（auth.guard）
- 认证装饰器（auth.decorator）
- 配置管理（auth.config）
- 异常过滤（auth.filter）
- Token 策略（strategies）
- 会话存储（stores）
- 权限引擎（permission）
- API 签名（signature）
- 审计日志（audit）
- Cookie 处理（cookie）
- 登录限流（rate-limit）
- 分布式锁（distributed-lock）
- 数据持久化（persistence）
- 密码加密（password）
- WebSocket 认证（ws）
- 国际化（i18n）
- 错误体系（errors）

#### 3.4.2 可选扩展模块（extras）

- OAuth2 / OIDC 授权服务器（oauth2）
- SSO 单点登录（sso）
- 微服务认证（microservice）

## 4 功能模块设计

### 4.1 认证服务模块

认证服务模块是本软件的核心，负责处理用户的登录、登出、Token 校验和会话生命周期管理。

#### 4.1.1 登录流程

1. 业务系统调用 `authService.login(userId, device, roles, permissions)` 方法。
2. 系统根据配置生成 JWT 或随机 Token。
3. 系统创建服务端 Session，保存用户标识、角色、权限、设备信息、登录时间等元数据。
4. 系统根据登录策略处理历史会话：单点登录策略下会踢掉旧会话；多点登录策略下保留所有会话；互斥登录策略下按设备类型保留一个会话。
5. 返回 Token 给调用方。

#### 4.1.2 登出流程

1. 调用方提供当前 Token。
2. 系统根据 Token 解析出会话标识。
3. 从 SessionStore 中删除对应会话。
4. 若启用了 Redis 分布式存储，同步清除相关索引。

#### 4.1.3 Token 校验流程

1. 请求到达时，AuthGuard 从请求头、Cookie 或 URL 参数中提取 Token。
2. 调用 TokenStrategy 校验 Token 签名或查询 Token 是否存在。
3. 从 SessionStore 中读取会话信息，确认会话未过期且未被踢出/封禁。
4. 若启用了设备指纹，校验请求 IP 和 User-Agent 是否与会话绑定信息一致。
5. 将用户信息注入请求上下文，供后续控制器使用。

### 4.2 会话存储模块

会话存储模块定义了统一的 `SessionStore` 接口，并提供内存和 Redis 两种实现。

#### 4.2.1 内存存储（MemoryStore）

- 使用 Map 结构保存会话数据。
- 采用 LRU 策略控制内存占用，支持最大会话数限制。
- 适用于单实例部署或开发测试环境。

#### 4.2.2 Redis 存储（RedisStore）

- 使用 Redis Hash 保存会话详情。
- 使用 Redis Set 建立用户与会话 ID 的索引关系，便于快速查询用户所有会话。
- 定期清理过期会话产生的僵尸索引，保证索引一致性。
- 适用于分布式部署和生产环境。

### 4.3 权限引擎模块

权限引擎模块实现了基于角色的访问控制（RBAC）。

#### 4.3.1 角色匹配

- 控制器通过 `@RequireRoles('admin', 'user')` 声明所需角色。
- AuthGuard 从会话中读取用户角色列表。
- 检查用户角色是否与所需角色存在交集。

#### 4.3.2 权限匹配

- 控制器通过 `@RequirePermissions('user:add', 'user:delete')` 声明所需权限。
- AuthGuard 从会话中读取用户权限列表。
- PermissionEngine 执行匹配，支持通配符模式，例如 `user:*` 可匹配 `user:add` 和 `user:delete`。
- 要求用户必须同时拥有所有声明的权限。

### 4.4 OAuth2 / OIDC 模块

OAuth2 / OIDC 模块实现了标准授权服务器功能。

#### 4.4.1 支持的授权模式

- authorization_code：授权码模式，支持 PKCE 扩展。
- password：资源所有者密码凭证模式。
- client_credentials：客户端凭证模式。
- refresh_token：刷新令牌模式。

#### 4.4.2 客户端管理

- 提供 OAuth2ClientStore 接口，支持内存和 Redis 实现。
- 客户端注册时需指定 clientId、clientSecret、redirectUris、授权范围、允许模式等。
- 客户端密钥比较采用 `crypto.timingSafeEqual` 防止时序攻击。

#### 4.4.3 Refresh Token Rotation

- 每次刷新访问令牌时，旧的 Refresh Token 被标记为已使用。
- 若检测到已使用的 Refresh Token 被再次使用，立即吊销该 Token Family 下所有令牌。
- 有效防止 Refresh Token 泄露后的持续滥用。

#### 4.4.4 OIDC 支持

- 提供 `/.well-known/openid-configuration` 端点。
- 支持签发 ID Token，包含 aud、nonce、exp 等标准校验字段。

### 4.5 SSO 单点登录模块

SSO 模块支持跨域单点登录场景。

#### 4.5.1 登录流程

1. 用户访问业务系统 A 的受保护资源，被重定向到 SSO 登录页。
2. 用户在 SSO 登录页完成认证，SSO 签发 Token。
3. 用户携带 Token 跳转回业务系统 A。
4. 业务系统 A 校验 Token 并建立本地会话。
5. 用户访问业务系统 B 时，系统自动识别 SSO Token，实现无需再次登录。

#### 4.5.2 Token 读取策略

支持从 HTTP Header、Cookie、URL Query 三种来源读取 Token，用户可按需配置优先级。

### 4.6 API 签名认证模块

API 签名模块用于保护开放接口，防止请求篡改和重放攻击。

#### 4.6.1 签名算法

采用 HMAC-SHA256 算法，签名原文格式为：

```
METHOD\nPATH\nTIMESTAMP\nNONCE\nBODY
```

请求方需提供 X-Signature、X-Timestamp、X-Nonce 三个请求头。

#### 4.6.2 重放防护

- 校验时间戳与服务器时间的偏差，默认允许 5 分钟误差。
- 使用 Nonce 去重机制，Redis 不可用时自动回退到内存 LRU。
- 签名比较采用常量时间比较函数，防止时序攻击。

### 4.7 登录限流模块

登录限流模块保护登录接口免受暴力破解和撞库攻击。

#### 4.7.1 限流维度

支持 IP 维度、账号维度、IP + 账号组合维度三种限流策略。

#### 4.7.2 限流算法

- 滑动窗口算法：在固定时间窗口内限制请求次数。
- 令牌桶算法：以固定速率补充令牌，桶满则丢弃。

#### 4.7.3 存储后端

支持内存实现（单机）和 Redis 实现（分布式）。

### 4.8 分布式锁模块

分布式锁模块用于保护高并发场景下的临界操作，如登录时的竞态条件。

#### 4.8.1 实现方式

- 内存实现：基于 Node.js 单进程内的锁管理。
- Redis 实现：基于 SET NX EX 命令获取锁，使用 Lua 脚本安全释放。

#### 4.8.2 使用场景

- 同一用户并发登录时的会话处理。
- OAuth2 Token 刷新时的竞态保护。

### 4.9 数据持久化模块

数据持久化模块为部分功能（如 OAuth2 客户端、用户持久化）提供统一的存储抽象。

#### 4.9.1 抽象接口

定义 PersistenceAdapter 接口，包含增删改查等基本操作。

#### 4.9.2 适配器实现

- SQLPersistenceAdapter：支持 MySQL、PostgreSQL、Oracle、SQLite。
- MongoDBPersistenceAdapter：支持 MongoDB。
- MemoryPersistenceAdapter：内存实现，适用于测试环境。

### 4.10 密码加密模块

密码加密模块提供安全的密码哈希存储能力。

#### 4.10.1 算法支持

- BCrypt：支持自定义 cost factor，默认不低于 12。
- Argon2：支持内存-hard 哈希，抵抗 GPU 暴力破解。

#### 4.10.2 使用方式

业务系统调用 PasswordEncoder 的 `encode` 和 `compare` 方法完成密码哈希和校验。

### 4.11 WebSocket 认证模块

WebSocket 认证模块为 Socket.IO 和原生 WebSocket 提供 Token 认证能力。

#### 4.11.1 Token 来源

优先从 `handshake.auth.token` 和 Authorization 请求头中读取，其次从 URL 查询参数中读取。

#### 4.11.2 认证流程

在 WebSocket 连接建立时执行与 HTTP 请求相同的 Token 校验逻辑，校验失败则拒绝连接。

### 4.12 审计日志模块

审计日志模块记录关键安全事件，支持 console、file、redis 三种存储后端。

#### 4.12.1 记录事件

- login、logout、force_logout、kick、ban
- switch_identity、renew
- open_safe_auth、close_safe_auth
- rpc_call、signature_auth

#### 4.12.2 日志内容

包括操作类型、用户标识、时间戳、IP 地址、操作结果等字段。

### 4.13 微服务认证模块

微服务认证模块为 NestJS 微服务架构提供鉴权能力。

#### 4.13.1 服务端

- AuthGuard.forMicroservice() 从 gRPC metadata 或 TCP 数据包中提取 Token。
- 支持 RPC IP 白名单校验。

#### 4.13.2 客户端

- MicroserviceAuthInterceptor 自动将当前请求的 Token 透传到下游 RPC 调用中。

### 4.14 国际化与错误码模块

- 按模块划分业务错误码，便于问题定位。
- 支持中文、英文两种语言的错误提示。
- 提供统一的 AuthException 异常体系。

## 5 接口设计

### 5.1 核心装饰器接口

| 装饰器 | 功能说明 |
|---|---|
| `@RequireLogin()` | 要求请求用户已登录 |
| `@RequireRoles(...roles)` | 要求用户具有指定角色之一 |
| `@RequirePermissions(...permissions)` | 要求用户具有全部指定权限 |
| `@RequireSafeAuth()` | 要求用户已完成二级认证 |
| `@RequireSignature()` | 要求请求通过 API 签名认证 |

### 5.2 核心服务接口

| 方法 | 功能说明 |
|---|---|
| `login(userId, device, roles, permissions)` | 用户登录，生成 Token 和会话 |
| `logout(token)` | 用户登出，销毁当前会话 |
| `validateToken(token)` | 校验 Token 并返回会话信息 |
| `forceLogout(sessionId)` | 强制下线指定会话 |
| `kickUser(userId, device?)` | 踢出指定用户（可按设备过滤） |
| `banUser(userId, duration)` | 封禁指定用户一段时间 |
| `renewSession(sessionId)` | 续签会话有效期 |
| `switchIdentity(userId, targetUserId, device)` | 切换当前身份 |
| `openSafeAuth(sessionId)` | 开启二级认证 |
| `closeSafeAuth(sessionId)` | 关闭二级认证 |

### 5.3 OAuth2 端点接口

| 端点 | 方法 | 功能说明 |
|---|---|---|
| `/oauth/authorize` | GET | 获取授权码 |

### 5.4 配置接口

本软件通过 `AuthModule.register(options)` 或 `AuthModule.registerAsync(options)` 进行配置。配置项包括：

- token：JWT 密钥和过期时间配置。
- storage：是否使用 Redis 及 Redis 连接选项。
- loginPolicy：登录策略、Token TTL、自动续签、同设备最大会话数、Remember Me TTL。
- permission：是否启用权限校验。
- safeAuth：二级认证开关和 TTL。
- sso：SSO 登录地址、Token 参数名、Token 读取策略。
- oauth2：OAuth2 功能开关。
- microservice：微服务认证开关和 RPC IP 白名单。
- audit：审计日志开关和存储后端。
- signature：API 签名密钥和时间戳容差。
- rateLimit：登录限流开关、算法、维度、阈值。
- fingerprint：设备指纹开关和模式。
- cookie：Cookie 名称、域、路径、安全属性等。
- distributedLock：分布式锁开关和重试参数。
- multiAccount：多账号切换开关和最大账号数。
- randomToken：随机 Token 的风格和前缀。

## 7 安全性设计

### 7.1 Token 安全

- JWT 密钥必须显式配置，默认密钥仅用于开发，生产环境会触发校验失败。
- 支持 Token 过期时间自定义，避免长期有效带来的风险。
- 支持 Refresh Token Rotation，检测重放使用后吊销整个 Token Family。

### 7.2 会话安全

- 支持设备指纹绑定，异常设备访问可触发严格拒绝或警告。
- 支持 Cookie 安全属性配置，默认启用 httpOnly、secure、sameSite。
- 支持 Token 自动续签，减少用户频繁登录的同时保证安全性。

### 7.3 密码安全

- 使用 BCrypt 或 Argon2 进行密码哈希，避免明文存储。
- BCrypt cost factor 默认不低于 12。
- 密码比较通过框架提供的 Encoder 完成，不直接暴露哈希逻辑。

### 7.4 接口安全

- API 签名使用 HMAC-SHA256，结合时间戳和 Nonce 防止重放。
- OAuth2 客户端密钥比较使用 `crypto.timingSafeEqual`，防止时序攻击。
- PKCE 仅支持 S256 方法，禁止 plain 方法。
- OAuth2 state 参数与授权码绑定，防止 CSRF 攻击。

### 7.5 限流与锁

- 登录接口支持 IP + 账号双维度限流，防止暴力破解。
- 分布式锁防止高并发登录产生的竞态条件。

### 7.6 审计与追溯

- 关键操作记录审计日志，支持 console、file、redis 三种后端。
- 审计内容包含操作类型、用户、时间、IP、结果，便于事后追溯。

}
